## Security Model

The Alloy ORM is built on a defense-in-depth approach to SQL injection prevention, using Temper's type system to enforce safety at compile time. The compiled Java output preserves all safety invariants.

### Defense Layers

**Layer 1: SafeIdentifier (CWE-89 -- table/column names)**

Table and column names must pass through `SrcGlobal.safeIdentifier()`, which validates against `[a-zA-Z_][a-zA-Z0-9_]*`. The internal `ValidatedIdentifier` class is not exported -- external code cannot construct one without validation. This is the only path to `appendSafe` at runtime, closing off identifier-based injection entirely.

```java
SrcGlobal.safeIdentifier("users");        // => ValidatedIdentifier (OK)
SrcGlobal.safeIdentifier("users; DROP");  // => bubble (rejected)
SrcGlobal.safeIdentifier("table--name");  // => bubble (rejected)
```

**Layer 2: Typed SqlPart hierarchy (CWE-89 -- value escaping)**

User values never touch `appendSafe`. Each type gets its own `SqlPart` subclass with type-appropriate rendering:

| Type | Rendering | Example |
|------|-----------|---------|
| `SqlString` | Quote-wraps with `'` escaping (`'` becomes `''`) | `'O''Brien'` |
| `SqlInt32` / `SqlInt64` | Bare integer via `.toString()` | `42` |
| `SqlBoolean` | Literal `TRUE` / `FALSE` | `TRUE` |
| `SqlFloat64` | Bare float (NaN/Infinity render as `NULL`) | `3.14` |
| `SqlDate` | Quote-wrapped date string with `'` escaping | `'2026-03-13'` |
| `SqlDefault` | Literal `DEFAULT` keyword | `DEFAULT` |

The `SqlPart` interface is sealed -- the compiler enforces exhaustive handling.

**Layer 3: SqlBuilder separation (CWE-89 -- structural safety)**

`SqlBuilder` enforces a clean separation between SQL structure and data:
- `appendSafe(str)` -- for SQL keywords and validated identifiers only
- `appendString(val)` -- wraps in `SqlString` (escaped)
- `appendInt32(val)` -- wraps in `SqlInt32` (bare integer)

There is no `appendRaw` or bypass method.

**Layer 4: Changeset pipeline (CWE-915 -- mass assignment)**

`Changeset` is a sealed interface -- `ChangesetImpl` is not exported, so construction only happens through the `SrcGlobal.changeset()` factory. The pipeline enforces:
- `cast(allowedFields)` -- field whitelisting via `List<SafeIdentifier>`, prevents mass assignment
- `validateRequired(fields)` -- non-nullable enforcement
- `toInsertSql()` / `toUpdateSql(id)` -- independently enforces non-nullable fields
- Raw `params` are never exposed -- only whitelisted `changes` are readable

**Layer 5: Query builder (CWE-89, CWE-400)**

`Query` requires `SafeIdentifier` for table names, column selections, ORDER BY clauses, and JOIN table names. WHERE and ON conditions accept only `SqlFragment` (never raw strings). `safeToSql(defaultLimit)` enforces result set bounds (CWE-400). `UpdateQuery` and `DeleteQuery` bubble on empty WHERE clauses, preventing accidental full-table mutations.

### Per-Phase Security Analysis

**JOIN Feature Security** -- JOIN keywords use a sealed `JoinType` interface with hardcoded strings. Table names require `SafeIdentifier`. ON conditions accept only `SqlFragment`. The `col()` helper produces qualified column references from two `SafeIdentifier` values joined by a hardcoded `.` separator.

**WHERE Clause Enrichment** -- `WhereClause` is sealed (`AndCondition`/`OrCondition`). Convenience methods (`whereNull`, `whereIn`, `whereBetween`, `whereLike`, `whereILike`) build fragments internally using only validated identifiers and typed values. Empty IN list produces `1 = 0`. LIKE/ILIKE patterns go through `SqlString` escaping.

**Aggregation** -- All 6 aggregate functions use hardcoded function names via `appendSafe`. Field names require `SafeIdentifier`. `selectExpr`, `groupBy`, `having`/`orHaving` reuse existing safe types. `distinct()` selects between two hardcoded strings.

**Set Operations and Subqueries** -- Set operation keywords (`UNION`, `INTERSECT`, `EXCEPT`) are hardcoded literals. Both operands are `Query` objects. Sub-SELECTs are parenthesized. `subquery()` alias requires `SafeIdentifier`. `existsSql()` wraps in hardcoded `EXISTS (...)`. `whereInSubquery()` combines `SafeIdentifier` field with `Query` subquery.

**Batch UPDATE/DELETE** -- `SetClause.field` requires `SafeIdentifier`. `SetClause.value` uses `SqlPart` (sealed, type-dispatched). `UpdateQuery` and `DeleteQuery` bubble on empty WHERE clauses, preventing accidental full-table mutations.

**Extended Query Features** -- `NullsPosition` and `CrossJoin` use sealed interfaces with hardcoded keywords. `LockMode` (`ForUpdate`/`ForShare`) uses sealed interfaces with hardcoded strings.

**Changeset Enrichment** -- All validation methods operate on the `changes` Map (string validation only, no SQL generation). Field parameters require `SafeIdentifier`. Values only reach SQL through `toInsertSql()`/`toUpdateSql()` which use type-dispatched escaping. Immutable pattern preserved.

**Schema Enrichment** -- `TableDef.primaryKey` uses `SafeIdentifier`. `pkName()` returns `SafeIdentifier.sqlValue` or hardcoded `"id"`. `FieldDef.defaultValue` uses `SqlPart` (sealed). `SqlDefault` renders hardcoded `"DEFAULT"`. Virtual fields are excluded from SQL entirely.

### Minimal Example

```java
import orm.src.SrcGlobal;
import orm.src.*;

public class Example {
    public static void main(String[] args) {
        // Define a table
        TableDef todosTable = new TableDef(
            SrcGlobal.safeIdentifier("todos"),
            java.util.List.of(
                new FieldDef(SrcGlobal.safeIdentifier("title"), new StringField(), false, null, false),
                new FieldDef(SrcGlobal.safeIdentifier("completed"), new IntField(), false, null, false)
            ),
            null
        );

        // SELECT query
        String selectSql = SrcGlobal.from(SrcGlobal.safeIdentifier("todos"))
            .orderBy(SrcGlobal.safeIdentifier("created_at"), true)
            .limit(50)
            .toSql()
            .toString();
        System.out.println(selectSql);
        // => SELECT * FROM todos ORDER BY created_at ASC LIMIT 50

        // INSERT via changeset
        var params = java.util.Map.of("title", "Buy milk", "completed", "0");
        var cs = SrcGlobal.changeset(todosTable, params)
            .cast(java.util.List.of(
                SrcGlobal.safeIdentifier("title"),
                SrcGlobal.safeIdentifier("completed")))
            .validateRequired(java.util.List.of(
                SrcGlobal.safeIdentifier("title")));

        if (cs.isValid()) {
            System.out.println(cs.toInsertSql().toString());
            // => INSERT INTO todos (title, completed) VALUES ('Buy milk', 0)
        }
    }
}
```

## Architecture

This repository is updated automatically by CI. The cascade works as follows:

```
alloy (Temper source)
  |
  | temper build -b java
  v
temper.out/java/
  |
  | CI publish (push to this repo)
  v
alloy-java (this repo)
  |
  | notify-app.yml workflow (triggered on push)
  v
alloy-java-app (demo application)
  |
  | vendors orm/, std/, temper-core/ into vendor/
  v
Application uses the ORM
```

A push to `main` in the [Alloy source repository](https://github.com/notactuallytreyanastasio/alloy) triggers CI, which builds the Temper compiler, compiles the ORM for all 6 backends, runs the test suite, and pushes the compiled Java output to this repository. The `notify-app.yml` workflow in this repo then automatically vendors the updated ORM into the [alloy-java-app](https://github.com/notactuallytreyanastasio/alloy-java-app) demo application.
